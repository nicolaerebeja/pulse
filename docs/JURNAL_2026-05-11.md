# Jurnal tehnic — Debugging notificări lipsă + înlocuire Tor cu Whisper
**Data:** 2026-05-11
**Durata:** ~3 ore

---

## 1. Problema raportată

Canalul `https://www.youtube.com/@Reuters/videos` publicase peste 10 video-uri de la pornirea bot-ului, dar au sosit doar 2 notificări.

---

## 2. Diagnosticare — Bug #1: `fetch_latest_video` lua un singur video

### Simptomul
Scheduler-ul detecta corect că există un video nou față de `last_video_id`, îl procesa, și actualiza `last_video_id` — dar toate video-urile intermediare publiciate între două poll-uri dispăreau definitiv.

### Cauza
`fetch_latest_video` folosea `maxResults=1` în YouTube Data API:

```python
_yt().playlistItems().list(
    part="snippet", playlistId=playlist_id, maxResults=1
).execute()
```

Dacă Reuters publica 5 video-uri între două poll-uri de 10 minute, scheduler-ul vedea doar video-ul #5 (cel mai recent), seta `last_video_id = #5`, iar video-urile #1–#4 erau **pierdute definitiv**.

### Fix
- Redenumit `fetch_latest_video` → `fetch_recent_videos`, returnează `list[dict]` în loc de `dict | None`
- `maxResults` crescut de la `1` la `_PLAYLIST_MAX = 10`
- `_poll_channel` rescris să proceseze **toate** video-urile noi față de `last_video_id`:

```python
ids = [v["video_id"] for v in recent]
if last_vid in ids:
    new_videos = recent[: ids.index(last_vid)]  # tot ce e mai nou decât last_vid
else:
    new_videos = recent  # all fetched — last_vid e prea vechi
```

- Procesare în ordine cronologică (`reversed(new_videos)`) — notificările sosesc oldest-to-newest
- `last_video_id` actualizat **după** procesarea tuturor video-urilor, nu înainte (cu `notif_sent` ca deduplicare, re-procesarea accidentală nu trimite duplicate)

## 3. Diagnosticare — Bug #2: Transcriptele eșuau silențios

Chiar după fix-ul #1, scheduler-ul detecta video-uri noi dar nu trimitea notificări. Din logs:

```
ERROR: [youtube] Q1ATJlO-ch8: Sign in to confirm you're not a bot.
_transcript_via_api failed: YouTube is blocking requests from your IP.
ensure_video_processed no transcript video=Q1ATJlO-ch8
```

Două erori distincte din două surse diferite.

### 3.1 Investigare yt-dlp: „Sign in to confirm you're not a bot"

Cookie-ul `GOOGLE_ABUSE_EXEMPTION` din `cookies.txt` era legat de IP-ul exit node-ului Tor cu care fusese generat (`192.42.116.55`). Circuitul Tor se rotise, IP-ul curent era `192.42.116.144`. YouTube refuza cookie-ul deoarece IP-ul nu mai corespundea.

### 3.2 Investigare youtube-transcript-api: „IP blocked"

Exit node-urile Tor sunt în mare parte IP-uri de datacenter — YouTube le blochează sistematic pentru `timedtext` API.

### Tentativă de fix: rotație circuit Tor

Am adăugat:
- `--ControlPort 9051` la pornirea Tor
- Funcție `_new_tor_circuit()` care trimite `SIGNAL NEWNYM` via socket la portul de control
- Retry de 3 ori cu circuit nou în `get_transcript()`

**Rezultat:** toate cele 3 încercări au eșuat identic — toate exit node-urile Tor testate sunt blocate de YouTube pentru acest tip de request.

### 3.3 Descoperirea reală: Reuters nu are subtitrări

Test direct fără Tor (IP RPi):

```python
api = YouTubeTranscriptApi()  # fara proxy
api.fetch('Q1ATJlO-ch8', languages=['en'])
# → "Subtitles are disabled for this video"
```

Din ultimele 10 video-uri Reuters verificate: **0 din 10 au subtitrări**. Reuters publică exclusiv footage brut — imagini cu voce de reporter sau mulțimi, fără transcriere automată YouTube.

Cele 2 notificări primite inițial au fost excepții — video-uri care accidental aveau `automatic_captions`.

---

## 4. Soluția finală: audio download + Whisper local

### Raționamentul

| Abordare | Risc IP block | Calitate | Complexitate |
|---|---|---|---|
| YouTube timedtext API prin Tor | Permanent blocat | N/A | Mare |
| YouTube timedtext API direct | Blocat (429 pe RPi) | N/A | — |
| Audio download (googlevideo.com CDN) direct | **Zero** | N/A | Mică |
| Whisper local pe audio descărcat | **Zero** (niciun apel YouTube) | Bună | Medie |

Concluzia: `googlevideo.com` CDN-ul nu aplică același bot detection ca `youtube.com` API. Audio download fără proxy funcționează imediat și rapid.

### Test de confirmare

```
yt-dlp Q1ATJlO-ch8 (fara Tor) → Q1ATJlO-ch8.webm (361 KB, ~1s)
faster-whisper base + int8 → transcriere în ~35s (model în memorie)
```

**Tor nu mai este necesar** — niciun component al pipeline-ului nu mai face request-uri la YouTube care să fie susceptibile de IP ban:
- YouTube Data API → autentificat cu API key (googleapis.com)
- Audio download → CDN `googlevideo.com`, fără bot detection
- Transcriere → locală, zero network calls la YouTube

## 5. Modificări implementate

### 5.1 `pulse/youtube.py` — rescris parțial

**Eliminat:**
- `_transcript_via_ytdlp()` — extragea URL caption via yt-dlp `process=False`, descărca via httpx prin Tor. Toată logica proxy/Tor.
- `_new_tor_circuit()` — rotație circuit Tor via control port.
- Import `httpx` (nu mai e folosit direct).

**Adăugat:**
- `_whisper_model` — singleton global; modelul se încarcă o singură dată la primul apel și rămâne în memorie.
- `_get_whisper()` — lazy loader pentru `WhisperModel("base", device="cpu", compute_type="int8")`.
- `_download_audio(video_id, tmpdir)` — descarcă best-quality audio stream cu yt-dlp fără proxy. Returnează calea fișierului descărcat.
- `_transcript_via_whisper(video_id)` — descarcă audio în `tempfile.mkdtemp()`, transcrie cu Whisper, șterge directorul temporar în `finally`.

**Modificat:**
- `get_transcript()` — flow nou: `_transcript_via_api()` (YouTube captions dacă există) → `_transcript_via_whisper()` (fallback universal).
- `_transcript_via_api()` — eliminat complet codul de proxy Tor; sesiunea `requests` merge direct.
- `fetch_recent_videos()` — era `fetch_latest_video()`, acum returnează `list[dict]` cu ultimele 10 video-uri.

### 5.2 `pulse/scheduler.py`

**Eliminat:**
- `import asyncio` (nu mai e necesar după eliminarea sleep-ului)
- `_TRANSCRIPT_FETCH_DELAY_S = 60` — sleep între fetch-uri exista pentru a evita IP ban; nu mai e necesar
- Logica `first_video` + `asyncio.sleep()` din bucla de procesare

**Modificat:**
- `_poll_channel()` — procesează lista `new_videos` fără sleep, în ordine cronologică

### 5.3 `pulse/__main__.py` — simplificat major

**Eliminat complet:**
- `_start_tor()` — pornea Tor ca subprocess, aștepta `Bootstrapped 100%`, ținea referința la proces, îl oprea în `finally`
- `_tor_process` — variabilă globală de subprocess
- Import `subprocess`, `time`

Fișierul a scăzut de la 82 la 50 de linii.

```

`tor` eliminat din `apt-get install`. `nodejs` păstrat (yt-dlp îl poate folosi pentru n-challenge la audio download).

```

Volum nou `whisper_cache` montat la `/root/.cache/huggingface` — modelul Whisper (~37MB int8 quantized) se descarcă o singură dată la primul start și persistă între restartări ale containerului.

## 6. Performanță măsurată pe RPi 5 (ARM64)

| Operație | Timp |
|---|---|
| Audio download (yt-dlp, ~200–400 KB) | ~1s |
| Încărcare model Whisper base int8 (prima dată în sesiune) | ~4s (din cache volum) |
| Transcriere clip 30s | ~15–25s |
| Transcriere clip 60s | ~30–50s |
| **Total per video (model în memorie)** | **~35–55s** |
| **Total primul video în sesiune** | **~90s** (include și download model la primul start absolut) |

La primul start absolut (volum `whisper_cache` gol): modelul se descarcă de pe HuggingFace (~37MB). Toate pornirile ulterioare folosesc cache-ul.

---

## 7. Comportament nou față de cel vechi

**Poll cu un singur video nou (cazul simplu):** identic cu înainte — 1 notificare.

**Poll cu N video-uri noi (Reuters tip):** se procesează toate N, notificările sosesc în ordine cronologică. Dacă N=5 și fiecare durează 40s → poll-ul durează ~200s. APScheduler cu `max_instances=1` (default) nu pornește un nou poll cât timp cel anterior rulează, deci nu există concurență.

**Video fără niciun speech detectabil** (b-roll, crowd noise): Whisper returnează text gol sau minim → `get_transcript` returnează `None` → `ensure_video_processed` returnează `None` → nicio notificare (comportament corect per REGULA 9).

## 9. Bug-uri descoperite în logs (sesiunea 2)

### Bug #3: `Can't parse entities` — notificări pierdute silențios

**Simptomul:** 2 notificări în cursul zilei au eșuat cu eroarea:
```
ERROR pulse.scheduler — notify failed chat=893120294 video=dMyp6aQ_fKE
error=Can't parse entities: can't find end of the entity starting at byte offset 1327
```

**Cauza:** Titlurile și rezumatele generate de LLM conțineau caractere speciale (`*`, `_`, backtick) care spărgeau parserul Markdown al Telegram.

**Fix:** Înlocuit `parse_mode="Markdown"` cu `parse_mode="HTML"` în tot codul, cu `html.escape()` aplicat pe orice text dinamic (titluri, rezumate, nume canale). Afectate: `scheduler.py` și `bot.py`.

### Bug #4: `ffmpeg could not be found` — video-uri HLS/m3u8 eșuau

**Simptomul:**
```
ERROR: m3u8 download detected but ffmpeg could not be found. Please install
WARNING pulse.youtube — _download_audio failed video=ChpShbMEH6w
```

Unele video-uri Reuters (live streams sau clipuri encodate HLS) necesitau ffmpeg pentru download — absent din imaginea Docker.

**Fix:** Adăugat `ffmpeg` în `apt-get install` din Dockerfile.

## 10. Funcționalitate nouă: notificări teaser + buton „Rezumat complet"

### Motivație
Notificările trimise conțineau rezumatul complet (titlu + 2-3 rânduri + bullets), ceea ce îngreuna scanarea rapidă a mai multor video-uri.

### Implementare

**`pulse/llm.py` — prompt redesenat:**
- Output structurat în două secțiuni separate de linia `---`
- Secțiunea 1 (teaser): 2-3 propoziții factuale — subiect, protagoniști, concluzie/eveniment central, cu cifre/nume exacte
- Secțiunea 2 (detaliat): titlu + 4-6 bullet points cu fapte precise
- Fără simboluri de formatare markdown în output (plain text + `•`)
- Nouă funcție `split_summary(text) -> tuple[str, str]` — separă robust teaser de detaliat, cu fallback dacă LLM-ul nu respectă formatul

**`pulse/scheduler.py` și `pulse/bot.py` — notificări schimbate:**
- Notificările (automate și la link manual) trimit acum **doar teaser-ul**
- Buton inline `📖 Rezumat complet` atașat fiecărei notificări
- Click pe buton → `edit_message_text` înlocuiește teaser-ul cu rezumatul detaliat (fără mesaj extra)
- Handler: `CallbackQueryHandler(show_full_summary, pattern=r"^full:")`
- Callback data format: `full:{video_yt_id}` (11 caractere, sub limita de 64 bytes)
