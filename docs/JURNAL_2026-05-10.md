## 1. Situația inițială — ce nu mergea și de ce

### Simptomele raportate
- Notificări goale, fără conținut
- Bot ignora comenzile sau executa altceva
- Ștergerea DB nu rezolva nimic — notificările veneau oricum
- Erori de IP blocat la extragerea transcripturilor YouTube

### Diagnosticul după citirea codului

#### 1.1 Nu exista mecanism de deduplicare reală
Colectorul YouTube lua ultimele 10 video-uri la fiecare rulare, indiferent dacă fuseseră văzute anterior. Nu exista nicio coloană `last_video_id` per canal. Rezultat: după orice ștergere a DB, toate video-urile recente erau re-procesate și re-trimise.

#### 1.2 Transcript gol → notificare trimisă oricum
Dacă `youtube-transcript-api` eșua (IP blocat → 429), `raw_transcript` era `None`. Codul nu oprea procesarea — continua, genera un summary gol, și trimitea notificarea cu body vid.

#### 1.3 `cmd_status` importa din proces separat
```python
from pulse.api.main import get_last_youtube_run  # bot.py
```
Bot-ul și API-ul rulau în procese Docker separate. Această funcție returna mereu `None` în bot — nu putea să acceseze memoria unui alt proces.

#### 1.4 Cod mort niciodată șters
Existau două implementări paralele ale aceluiași lucru:
- `bot/notify.py` — versiunea nouă, procedurală
- `bot/notifier.py` — versiunea veche cu `structlog` (interzis în CLAUDE.md) și ORM (interzis)

Ambele existau în același repo. Confuzie totală despre care se executa.

#### 1.5 `embed_item` chemat pentru fiecare video
Pipeline-ul apela embedding la Ollama după fiecare video procesat. Ollama era dezactivat → erori silențioase la fiecare item.

#### 1.6 Același canal procesat de N ori
Dacă 2 utilizatori urmăreau același canal, transcriptul era descărcat de două ori, independent, fără nicio partajare.

#### 1.7 30+ fișiere pentru 3 funcționalități
```
collectors/base.py, collectors/youtube.py, collectors/youtube_transcript.py,
collectors/facebook.py, collectors/instagram.py, collectors/linkedin.py,
collectors/twitch.py, collectors/url_ingester.py,
processing/pipeline.py, processing/summarize.py, processing/summarizer.py,
processing/embed.py, processing/embedder.py,
llm/base.py, llm/deepseek.py, llm/deepseek_client.py, llm/prompts.py,
bot/main.py, bot/commands.py, bot/callbacks.py, bot/telegram.py,
bot/notify.py, bot/notifier.py,
portal/app.py, portal/pages/*, portal/components/*,
siyuan/client.py, rag/*, core/logging.py, ...
```
Imposibil de depanat fără să petreci 30 de minute să înțelegi flow-ul.

## 2. Decizia — rewrite complet

### De ce nu am reparat codul vechi
Problemele erau structurale, nu punctuale. A repara fișier cu fișier ar fi însemnat să menții o arhitectură greșită. Mai simplu și mai sigur: șterge tot și rescrie cu reguli clare.

### Principiul de bază al rescrierii
**Un developer care vede codul pentru prima dată trebuie să înțeleagă flow-ul complet în sub 10 minute.**

## 3. Regulile create (anti-overengineering)

Înainte de a scrie cod, am creat două documente de reguli:

### `docs/REGULI.md` — Constituția Anti-Overengineering
10 reguli concrete:
- **REGULA 1**: Nicio abstracție preventivă — creezi funcție/modul doar când ai 3+ cazuri reale
- **REGULA 2**: Copiază înainte să abstractizezi (2 locuri → copy-paste, 3+ → extrage funcție)
- **REGULA 3**: Funcții liniare, max 2 niveluri de imbricare, max 50 linii
- **REGULA 4**: Nume care descriu acțiunea (`resolve_channel`, nu `handleData`)
- **REGULA 5**: Zero pattern enterprise — fără ABC, DI, Repository, Service layer, factories
- **REGULA 6**: Raw SQL exclusiv — ORM interzis în cod de aplicație
- **REGULA 7**: Un singur proces — bot + scheduler în același asyncio
- **REGULA 8**: Comentariile explică DE CE, nu CE
- **REGULA 9**: Erori clare — niciodată `except Exception: pass`; transcript gol = nicio notificare
- **REGULA 10**: Zero hardcoding — totul în `config.py` via pydantic-settings

### `docs/MVP.md` — Scope exact
Ce face MVP-ul și, la fel de important, ce **nu** face:

| Exclus | Motivul |
|---|---|
| Portal web (NiceGUI) | Zero necesitate demonstrată |
| RAG / embeddings | Complexitate fără beneficiu imediat |
| SiYuan integration | Overhead de documentare automată |
| Facebook/Instagram/LinkedIn/Twitch | MVP = YouTube only |
| FastAPI separată | Un singur proces e suficient |
| Feedback like/dislike | Neesențial |

## 4. Arhitectura nouă — ce s-a construit

### Structura fișierelor (7 surse)
```
pulse/
├── __main__.py     # entrypoint: Tor + bot + scheduler
├── config.py       # pydantic-settings din .env
├── db.py           # engine + fetch_all/one/value/execute
├── models.py       # SQLAlchemy models doar pentru Alembic
├── youtube.py      # resolve_channel, fetch_latest_video, get_transcript
├── llm.py          # summarize(transcript) -> str
├── scheduler.py    # poll_all_channels() la N minute
└── bot.py          # toți handlerii Telegram
```

Față de ~35 de fișiere anterioare → **7 fișiere sursă**.

### Schema DB nouă (5 tabele)

```sql
chats           -- chat_id BIGINT PK, chat_type, title
                -- chat_id poate fi chat privat SAU grup Telegram

yt_channels     -- id SERIAL PK, yt_id TEXT UNIQUE, name, last_video_id, checked_at
                -- last_video_id = cheia de deduplicare

chat_subs       -- chat_id → yt_channel_id (many-to-many)
                -- un chat urmărește N canale

videos          -- id SERIAL PK, yt_id TEXT UNIQUE, title, transcript, summary
                -- partajat între toți utilizatorii care urmăresc același canal
                -- transcript/summary calculat o singură dată per video

notif_sent      -- chat_id + video_yt_id (PK compus)
                -- deduplicare simplă: dacă există rândul, nu mai trimite
```

Față de schema veche (8 tabele cu UUID-uri, relații complexe, RAG tables, app_logs ORM).

### Proprietăți cheie ale designului

**`last_video_id` per canal:**
La fiecare poll, comparăm ultimul video de pe canal cu `last_video_id` stocat. Dacă sunt identice → nimic nou. Dacă diferă → video nou → procesare. La prima verificare (când `last_video_id` este NULL), doar salvăm ID-ul fără să notificăm — evităm spam la adăugarea unui canal nou.

**`videos` partajat:**
Dacă 10 chat-uri urmăresc același canal, transcriptul se descarcă o singură dată și summary-ul se generează o singură dată. Toate cele 10 chat-uri primesc notificarea, dar procesarea se face o singură dată.

**"Grupuri" = chat Telegram nativ:**
Nu am implementat niciun concept de grup în baza de date. Un grup Telegram este pur și simplu un `chat_id` cu `chat_type = "group"`. Bot-ul funcționează identic în chat privat și în grup. Abonamentele și notificările se leagă de `chat_id`, indiferent dacă e privat sau grup. Zero logică extra în cod.

**Un singur proces:**
Bot-ul Telegram și scheduler-ul APScheduler rulează în același proces `asyncio`. Nu există comunicare inter-proces, nu există 2 containere de aplicație, nu există FastAPI separată.

## 5. Problemele întâmpinate în implementare și cum s-au rezolvat

### 5.1 Migrarea veche blocata baza de date
**Problema:** La primul `alembic upgrade head`, eroare:
```
FAILED: Can't locate revision identified by '0002'
```
DB-ul păstra înregistrarea versiunii `0002` care fusese ștearsă.

**Fix:** `docker compose down -v` — ștergere completă volum postgres + recreare de la zero cu noua schemă.

### 5.2 yt-dlp: n challenge solving failed
**Problema:** yt-dlp nu putea rezolva "n challenge" YouTube (mecanism JS de obfuscare URL-uri). Returna doar storyboard-uri (imagini), nicio subtitrare, nicio explicație utilă în logs.

**Investigare:** `yt-dlp --verbose --list-subs` a arătat:
```
JS runtimes: none
[youtube] JS Challenge Providers: bun (unavailable), deno (unavailable),
          node (unavailable), quickjs (unavailable)
```
Lipsea un JavaScript runtime în container.

**Fix:** `apt-get install nodejs` în Dockerfile.

**Problemă nouă:** Chiar și după instalarea Node.js, yt-dlp raporta `node (unavailable)`. Cauza: yt-dlp folosea `extract_info()` cu `process=True` (default) care ridica excepție `Requested format is not available` înainte să returneze info-ul cu URL-urile de subtitrare.

**Fix final:** `extract_info(url, download=False, process=False)` — returnează metadata raw fără format selection, deci fără excepție. Subtitrările sunt prezente în `automatic_captions` chiar dacă video formatele nu sunt disponibile.

### 5.3 IP blocat de YouTube pentru timedtext API
**Problema:** Chiar dacă yt-dlp putea extrage URL-ul subtitrării (în format `json3`), descărcarea URL-ului returna `429 Too Many Requests`. Același 429 apărea și de pe host, direct cu `curl`, și în container — confirmat că e o blocare la nivel de IP.

**Ce nu a funcționat:**
- Cookies din browser (exportate cu "Get cookies.txt LOCALLY") → 429 în continuare
- User-Agent de browser real → 429
- `android` client extractor în yt-dlp → format tot indisponibil
- `requests.Session` cu cookies → 429

**Investigare:** `curl` direct de pe RPi (fără Docker) → tot 429. Problema nu e Docker, e IP-ul extern al serverului blocat de YouTube.

**Fix:** **Tor** ca proxy SOCKS5 pentru toate requesturile YouTube.

- Tor instalat în container via `apt-get install tor`
- Pornit în `__main__.py` înainte de bot, cu așteptare `Bootstrapped 100%`
- yt-dlp configurat cu `proxy: socks5://127.0.0.1:9050`
- `httpx` configurat cu același proxy pentru descărcarea json3
- `requests.Session` configurat cu proxy pentru youtube-transcript-api fallback

**Rezultat:** Prin Tor, timedtext API returnează `200 OK`.

### 5.4 Tor image incompatibil cu ARM64
**Problema:** Prima tentativă a folosit `peterdavehello/tor-socks-proxy` ca serviciu Docker separat. Imaginea e compilată pentru amd64, RPi-ul e ARM64 → container în crash loop.

**Fix:** Tor instalat direct în containerul `pulse` (nu serviciu separat), pornit ca subprocess la startup.

### 5.5 `youtube-transcript-api` — constructor incompatibil
**Problema:** Cod scris cu `YouTubeTranscriptApi(cookies=cookies_file)` → `got an unexpected keyword argument 'cookies'`. Versiunea instalată nu suportă `cookies` în constructor.

**Fix:** API-ul acceptă `http_client` (un `requests.Session`). Cookies încărcate într-un `MozillaCookieJar` și atribuite session-ului.

### 5.6 `cookies.txt` montat read-only
**Problema:** yt-dlp încearcă să **scrie** în fișierul de cookies (îl actualizează cu valori noi). Fișierul era montat cu `:ro` în compose.yaml → `[Errno 30] Read-only file system`.

**Fix:** Eliminat `:ro` din mount.

## 7. Infrastructura finală

```
Docker Compose:
  postgres   — pgvector/pgvector:pg16
  pulse      — python:3.11-slim + nodejs + tor

pulse container:
  PID 1: python -m pulse
    ├── subprocess: tor (SOCKS5 pe 127.0.0.1:9050)
    ├── APScheduler: poll_all_channels() la 10 minute
    └── python-telegram-bot: polling Telegram API
```

### Flow complet pentru un video nou

```
APScheduler (10 min)
  → fetch_latest_video(channel_yt_id)       # YouTube Data API v3
  → compară cu last_video_id din DB
  → dacă nou:
      → get_transcript(video_id)
          → yt-dlp process=False → URL json3
          → httpx GET url via Tor → json3 text
          → fallback: youtube-transcript-api via Tor
      → summarize(transcript)               # DeepSeek API
      → INSERT INTO videos (shared)
      → UPDATE yt_channels SET last_video_id
      → pentru fiecare chat abonat:
          → verifică notif_sent (dedup)
          → bot.send_message(chat_id, summary)
          → INSERT INTO notif_sent
```

### Flow pentru link video trimis în bot

```
Utilizator trimite "https://youtube.com/watch?v=..."
  → handler detectează video ID
  → verifică videos table (cache)
  → dacă există summary → răspunde instant
  → dacă nu:
      → get_transcript via Tor
      → summarize via DeepSeek
      → INSERT INTO videos
      → răspunde cu summary
```

## 8. Fișiere de configurare relevante

```
DATABASE_URL=postgresql+asyncpg://pulse:...@postgres:5432/pulse
TELEGRAM_BOT_TOKEN=...
YOUTUBE_API_KEY=...
LLM_API_KEY=...
LLM_BASE_URL=https://api.deepseek.com
LLM_MODEL=deepseek-chat
YOUTUBE_COOKIES_FILE=/data/cookies.txt
YOUTUBE_PROXY=socks5://127.0.0.1:9050
POLL_INTERVAL_MINUTES=10
```

### `cookies.txt`
Fișier Netscape cookies exportat din Chrome cu extensia "Get cookies.txt LOCALLY" pe youtube.com. Montat în container la `/data/cookies.txt`. Folosit de yt-dlp și requests pentru autentificare YouTube.
